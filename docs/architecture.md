# 王者之奕 - 架构文档

## 概述

王者之奕是一个生产级的自走棋游戏服务端实现，采用 Python + FastAPI 技术栈，支持 8 人在线对战。

## 核心模块

### 1. 应用入口 (main.py)

FastAPI 应用入口，负责：
- 应用生命周期管理
- 路由注册
- 中间件配置
- CORS 设置

```python
app = FastAPI(
    title="王者之奕",
    description="自走棋游戏服务器",
    version="0.1.0",
    lifespan=lifespan,
)
```

### 2. API 模块 (api/)

RESTful HTTP API，提供：
- 玩家信息查询/更新
- 段位/统计查询
- 背包管理
- 玩家搜索

### 3. WebSocket 模块 (ws/)

WebSocket 实时通信：
- 连接管理（建立、断开、重连）
- 消息路由
- 心跳检测
- 会话管理

### 4. 游戏模块 (game/)

游戏核心逻辑：

#### 4.1 英雄池 (hero_pool.py)
- **SharedHeroPool**: 8人共享英雄池
- **HeroFactory**: 英雄实例创建
- **ShopManager**: 商店管理

#### 4.2 羁绊系统 (synergy.py)
- **SynergyManager**: 羁绊计算与激活
- 支持 9 种种族、8 种职业羁绊

#### 4.3 经济系统 (economy.py)
- **EconomyManager**: 经济管理
- 连胜/连败奖励
- 利息计算
- 等级系统

#### 4.4 战斗系统 (battle/simulator.py)
- **BattleSimulator**: 确定性战斗模拟
- 帧同步支持
- 技能释放

### 5. 房间模块 (room/)

#### 5.1 GameRoom (game_room.py)
游戏房间状态机：

```
WAITING → PREPARING → BATTLING → SETTLING → (循环)
                        ↓
                    FINISHED
```

状态说明：
- **WAITING**: 等待玩家加入
- **PREPARING**: 准备阶段，购买/调整阵容
- **BATTLING**: 战斗阶段，自动战斗
- **SETTLING**: 结算阶段，计算伤害
- **FINISHED**: 游戏结束

#### 5.2 RoomManager (manager.py)
单例模式房间管理器：
- 创建/销毁房间
- 玩家加入/离开
- 房间查找与匹配
- 空闲房间清理

### 6. 匹配模块 (match/)

#### 6.1 匹配队列 (queue.py)
- 按段位分队列
- FIFO 匹配
- 扩大搜索范围机制
- AI 填充支持

#### 6.2 段位系统 (rating.py)
- 8 级段位体系
- 星星升降
- 定位赛

#### 6.3 ELO 算法 (elo.py)
- ELO 积分计算
- 胜率预测

### 7. 数据模型 (models/)

SQLAlchemy ORM 模型：
- **PlayerDB**: 玩家账户
- **PlayerRankDB**: 段位数据
- **PlayerStatsDB**: 统计数据
- **HeroConfigDB**: 英雄配置
- **MatchRecordDB**: 对局记录

### 8. 通信协议 (shared/protocol/)

#### 8.1 消息定义 (messages.py)
Pydantic 消息模型：
- 连接消息（connect/reconnect）
- 房间消息（join/leave/ready）
- 商店消息（refresh/buy/sell）
- 英雄消息（place/move/upgrade）
- 游戏消息（round/battle/game_over）

#### 8.2 编解码 (codec.py)
- JSON 序列化/反序列化
- 消息类型路由

## 数据流

### 1. 玩家连接流程

```
Client                     Server
  │                          │
  │──── connect ────────────>│
  │                          │ 验证 token
  │                          │ 创建 Session
  │<─── connected ───────────│
  │                          │
  │──── heartbeat ──────────>│ (每30秒)
  │<─── heartbeat_ack ───────│
  │                          │
```

### 2. 游戏流程

```
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│ WAITING │───>│PREPARING│───>│BATTLING │───>│SETTLING │
└─────────┘    └─────────┘    └─────────┘    └─────────┘
     │              │              │              │
     │              │              │              │
     │         分发金币        战斗配对        计算伤害
     │         刷新商店        执行战斗        更新血量
     │         玩家操作        同步状态        检查淘汰
     │              │              │              │
     │              └──────────────┴──────────────┘
     │                     (循环直到结束)
     │                           │
     └────────────────────────>│
                          FINISHED
```

### 3. 战斗同步

```
Server                           Clients
  │                                │
  │─── battle_start ──────────────>│
  │                                │
  │    [模拟战斗帧]                 │
  │                                │
  │─── battle_sync ───────────────>│ (每帧/关键帧)
  │─── battle_event ──────────────>│ (伤害/技能/死亡)
  │                                │
  │─── battle_result ─────────────>│
  │                                │
```

## 技术选型

| 组件 | 技术 | 说明 |
|------|------|------|
| Web 框架 | FastAPI | 高性能异步框架 |
| ORM | SQLAlchemy 2.0 | 异步 ORM |
| 数据库 | MySQL | 主数据库 |
| 缓存 | Redis | 会话/排行榜缓存 |

| 日志 | structlog | 结构化日志 |
| ASGI | Uvicorn | ASGI 服务器 |

## 扩展性

### 水平扩展

1. **无状态服务**: HTTP API 可无状态扩展
2. **房间隔离**: 房间间无依赖，可分布部署
3. **Redis 共享**: 会话和缓存共享

### 垂直扩展

1. **异步 IO**: 高并发处理
2. **连接池**: 数据库连接复用
3. **缓存策略**: 减少数据库压力

## 安全考虑

1. **Token 认证**: 所有操作需验证身份
2. **房间隔离**: 玩家只能操作自己房间
3. **输入验证**: Pydantic 严格类型检查
4. **SQL 注入防护**: ORM 参数化查询

## 性能优化

1. **帧同步**: 减少网络传输量
2. **差异同步**: 只发送变化数据
3. **共享英雄池**: 内存中单例
4. **异步处理**: 非阻塞 IO
