# 代码审核报告

**审核时间**: 2026-03-01 15:00
**审核范围**: wangzhe-chess 项目
**工具**: Ruff (代码风格), MyPy (类型检查)

## 审核结果

### 🔴 严重问题 (必须修复)

#### 1. WebSocket 消息处理器集成不完整
- **文件**: `src/server/main.py`, `src/server/ws/handler.py`
- **问题描述**: 多个 WebSocket 处理器模块（emote_ws, lineup_ws, spectator_ws 等）没有被正确注册到主处理器，导致消息无法路由
- **影响**: 游戏功能（商店、表情、阵容等）无法正常工作
- **建议**: 
  - 统一处理器注册机制
  - 避免循环导入问题
  - 使用消息类型枚举作为字典 key

#### 2. 导入路径不一致
- **文件**: `src/server/ws/equipment_ws.py`, `src/server/ws/voting_ws.py`
- **问题描述**: 混用相对导入和绝对导入
  ```python
  # 错误
  from src.server.game.equipment import ...
  # 正确
  from ..game.equipment import ...
  ```
- **影响**: 代码可移植性差，容易出现导入错误
- **建议**: 统一使用相对导入（相对于 src/ 目录）

#### 3. 未使用的导入和变量
- **文件**: 多个文件
- **问题数**: 158 (F401 unused-import)
- **影响**: 代码冗余，影响可读性
- **建议**: 清理未使用的导入

#### 4. 类型注解问题
- **文件**: 多个文件
- **问题数**: 11 (F841 unused-variable)
- **影响**: 运行时可能出错的未使用变量
- **建议**: 移除未使用的变量

#### 5. 导入顺序不规范
- **文件**: 79 个文件 (I001 unsorted-imports)
- **影响**: 代码可读性差
- **建议**: 运行 `ruff check --fix` 自动修复

### 🟡 警告 (建议修复)

#### 1. 类型检查错误
- **文件**: 多个文件
- **问题**: 
  - 返回类型不匹配 (no-any-return)
  - 缺少类型参数 (type-arg)
  - 赋值类型不兼容 (return-value, arg-type)
- **示例**:
  ```python
  # equipment.py:70
  # 返回 None 但期望 SpecialEffectType
  def get_effect_type(self) -> SpecialEffectType:
      return None  # ❌
  ```
- **建议**: 
  - 修复类型注解
  - 添加 Optional[T] 处理可能为 None 的情况
  - 使用泛型类型参数 (list[int] 而不是 list)

#### 2. 过时的类型注解
- **文件**: 129 个文件
- **问题**: 使用 typing.Dict, typing.List, typing.Set 而不是内置类型
- **建议**: 迁移到 Python 3.9+ 内置类型 (dict, list, set)
  ```python
  # 旧写法
  from typing import Dict, List
  # 新写法（Python 3.9+）
  data: dict[str, int] = {}
  items: list[int] = []
  ```

#### 3. 代码格式问题
- **文件**: 多个文件
- **问题**: 
  - 行尾空白 (W293): 15 个
  - 尾随空格 (W291): 1 个
- **建议**: 运行 `ruff format .` 自动修复

#### 4. 函数缺少类型注解
- **文件**: 多个文件
- **问题**: 函数/方法缺少返回类型注解
- **建议**: 添加类型注解提高代码可维护性

### 🟢 优化建议

#### 1. 性能优化
- **WebSocket 消息路由**: 当前使用字典查找，可以考虑使用 match-case (Python 3.10+) 提高可读性和性能
- **数据库查询**: 检查是否有 N+1 查询问题
- **缓存**: 对于频繁查询的数据（如英雄池、羁绊配置）考虑添加缓存

#### 2. 代码结构优化
- **模块职责**: src/server/ws/ 目录下有多个处理器文件，但集成方式混乱
  - 建议: 创建一个清晰的注册机制
  - 建议: 使用依赖注入而不是全局变量
- **错误处理**: 统一错误处理机制

#### 3. 安全性
- **输入验证**: WebSocket 消息的 decode_message 已经有验证，但需要检查业务逻辑层的验证
- **SQL 注入**: 使用 SQLAlchemy ORM，基本安全，但需要检查原生 SQL 查询
- **权限控制**: WebSocket 连接后没有额外的权限验证

#### 4. 测试覆盖率
- **状态**: 测试文件存在，但覆盖率未知
- **建议**: 运行 `pytest --cov` 查看覆盖率
- **目标**: 核心逻辑覆盖率 > 80%
