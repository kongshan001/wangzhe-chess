# Test With Code

## 编号
R-004

## 描述
任何代码变更必须伴随相应的测试。测试不是可选项，而是代码交付的组成部分。

具体要求:

**新功能开发:**
- 必须创建新的测试文件或在现有测试文件中添加测试用例
- 测试文件路径遵循约定: `test/` 或 `__tests__/` 目录下，文件名与源文件对应
- 测试必须覆盖正常路径（happy path）和至少一个边界情况
- 测试文件必须能通过 `npm test`、`pytest` 或项目配置的测试命令执行

**Bug 修复:**
- 必须添加回归测试，验证 bug 已被修复
- 回归测试应模拟原始 bug 的触发条件
- 回归测试必须在修复前失败、修复后通过

**重构:**
- 重构前必须确保现有测试全部通过
- 重构后运行完整测试套件，确保行为不变
- 如重构改变了公共接口，须同步更新测试

测试覆盖率要求:
- 新增代码的测试覆盖率不低于 80%
- 关键路径（核心逻辑、数据处理、边界处理）的测试覆盖率不低于 90%
- 测试代码本身也需要清晰、可维护

## 检查方法
QA agent 在评估阶段进行检查:

1. 检查 worktree 中是否存在测试文件（匹配 `test/`、`__tests__/`、`*.test.*`、`*.spec.*` 等模式）
2. 对比新增/修改的源文件与对应测试文件，确认覆盖关系
3. 检查测试是否能成功执行（通过项目配置的测试命令）
4. 评估测试质量: 是否覆盖正常路径、边界情况、错误处理
5. 对 bug 修复，检查是否包含回归测试

QA agent 在 `qa_report.json` 的 `test_completeness` 评分维度中反映测试覆盖情况。

## 违反后果
QA agent 检测到测试缺失时，`test_completeness` 评分维度将被降低（通常不超过 5/10）。这可能导致整体评分未通过阈值，触发 hot iteration（热迭代）要求补全测试。严重情况下（如核心功能完全无测试），QA agent 可直接给出 fail 评价，强制进入 hot iteration。
